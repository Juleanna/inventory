1. Що таке CI/CD і навіщо це потрібно

CI — Continuous Integration (безперервна інтеграція)

Автоматична перевірка кожної зміни в коді: лінтинг, тестування, збірка. Кожен push або pull request запускає набір перевірок, які гарантують, що новий код:

  • Відповідає стандартам форматування (Black, isort, ESLint)
  • Не містить синтаксичних помилок та bad practices (flake8, TypeScript)
  • Проходить всі тести (pytest, Vitest)
  • Успішно збирається в продакшн-бандл

CD — Continuous Delivery (безперервна доставка)

Автоматичне розгортання після успішного проходження всіх перевірок. При push у main пайплайн автоматично:

  • Збирає Docker-образи backend та frontend
  • Публікує їх у GitHub Container Registry (ghcr.io)
  • Деплоїть на продакшн-сервер через SSH

Переваги

Якість коду
Помилки виявляються автоматично ще до merge. Жоден баг не потрапить у main без проходження перевірок.
Єдиний стандарт
Всі розробники дотримуються однакових правил форматування та стилю коду.
Швидкий зворотний зв'язок
Розробник бачить результат перевірок за 2-5 хвилин після push.
Безпечний merge
Pull Request можна влити тільки якщо всі перевірки пройшли (при налаштуванні branch protection).

2. Архітектура пайплайна

Пайплайн складається з 8 jobs (задач), які виконуються в 4 етапи:

Етап 1
CI: перевірки
backend-lint
backend-test
frontend-lint
frontend-test
↓
Етап 2
CI: збірка
frontend-build
↓ тільки main
Етап 3
CD: Docker
docker-backend
docker-frontend
↓
Етап 4
CD: деплой
deploy
Етапи 3-4 (CD) запускаються тільки при push у main. Pull Request'и та push у develop виконують тільки CI-перевірки (етапи 1-2).
Job Тип Інструменти Час
backend-lint Лінтинг Black, flake8, isort ~30с
backend-test Тести pytest + coverage ~1-2хв
frontend-lint Лінтинг ESLint + TypeScript ~30с
frontend-test Тести Vitest ~30с
frontend-build Збірка tsc + Vite ~1хв
docker-backend Docker Buildx + GHCR ~2-3хв
docker-frontend Docker Buildx + GHCR ~1-2хв
deploy Деплой SSH + Docker Compose ~1хв

3. Тригери запуску

Пайплайн запускається автоматично при двох подіях:

Подія Гілки Коли спрацьовує
push main, develop Кожен push (комміт) у ці гілки
pull_request main Створення або оновлення PR в main

Конфігурація тригерів

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

Як змінити тригери

Додати нову гілку для перевірки:

on:
  push:
    branches: [main, develop, feature/*]  # всі feature-гілки

Запускати тільки при зміні певних файлів:

on:
  push:
    branches: [main]
    paths:
      - 'inventory_project/**'   # тільки при зміні бекенду
      - 'frontend/**'             # тільки при зміні фронтенду

Запуск вручну (workflow_dispatch):

on:
  workflow_dispatch:  # кнопка "Run workflow" у GitHub

4. Опис кожного Job

4.1. backend-lint Лінтинг

Перевіряє Python-код на відповідність стандартам форматування.

Black — автоформатер коду

Перевіряє, що весь Python-код відформатовано згідно з PEP 8 (з деякими відхиленнями Black). Параметр --check не змінює файли, а лише повідомляє про невідповідність.

black --check inventory_project/ --exclude migrations

flake8 — статичний аналізатор

Знаходить помилки стилю, невикористані імпорти, надто довгі рядки тощо. Конфігурація в setup.cfg.

flake8 inventory_project/
# Конфігурація (setup.cfg):
# max-line-length = 120
# exclude = migrations,__pycache__,venv,.venv
# extend-ignore = E203  (конфлікт з Black)

isort — сортування імпортів

Перевіряє порядок імпортів: стандартна бібліотека → третя сторона → Django → DRF → локальні модулі.

isort --check-only inventory_project/ --skip migrations
# Конфігурація (setup.cfg):
# profile = black
# sections = FUTURE,STDLIB,THIRDPARTY,DJANGO,DRF,FIRSTPARTY,LOCALFOLDER
Чому E203 ігнорується? Black форматує слайси як data[start : end] з пробілами, а flake8 вважає це помилкою E203. Ці інструменти конфліктують, тому E203 додано в extend-ignore.

4.2. backend-test Тести

Запускає Django-тести з реальною базою даних PostgreSQL та Redis.

Сервіси (Service Containers)

GitHub Actions піднімає Docker-контейнери для тестів:

Сервіс Образ Порт Призначення
PostgreSQL postgres:16 5432 Тестова база даних
Redis redis:7 6379 Кеш / брокер Celery

Змінні середовища

SECRET_KEY=test-secret-key-for-ci
DB_NAME=inventory_test
DB_USER=postgres
DB_PASSWORD=postgres
DB_HOST=localhost
DB_PORT=5432
DEBUG=true

Кроки виконання

  1. Встановлення системних залежностей для python-ldap: libldap2-dev, libsasl2-dev
  2. Встановлення Python 3.12 з кешуванням pip
  3. Встановлення залежностей: pip install -r requirements.txt
  4. Виконання міграцій: python manage.py migrate
  5. Запуск тестів: python -m pytest --cov=inventory --cov-report=xml -v

Покриття коду (Coverage)

Прапорці --cov=inventory та --cov-report=xml генерують звіт покриття у форматі XML. Цей файл можна завантажити в Codecov, SonarCloud або інші сервіси аналітики.

4.3. frontend-lint Лінтинг

Перевіряє TypeScript/React код за допомогою ESLint.

Конфігурація ESLint

Файл frontend/eslint.config.js використовує flat config:

  • @eslint/js — базові правила JavaScript
  • typescript-eslint — правила TypeScript (невикористані змінні, any тощо)
  • eslint-plugin-react-hooks — правила React hooks (правила залежностей)
  • eslint-plugin-react-refresh — правила React Fast Refresh (HMR)
cd frontend && npm run lint
# Еквівалент: eslint . --ext .ts,.tsx

4.4. frontend-test Тести

Запускає юніт-тести фронтенду через Vitest.

cd frontend && npx vitest run
# run — одноразовий запуск (не watch-режим)

4.5. frontend-build Збірка

Фінальна перевірка: чи збирається проєкт без помилок.

cd frontend && npm run build
# Виконує: tsc -b && vite build
# 1. tsc -b — TypeScript компіляція (перевірка типів)
# 2. vite build — продакшн бандл
needs: [frontend-lint, frontend-test] — цей job запускається тільки після успішного завершення лінтингу та тестів. Якщо хоча б один з них впав, збірка не починається.

4.6. docker-backend Docker

Збирає Docker-образ backend та публікує в GitHub Container Registry.

Умова запуску: тільки при push у main, після успішного backend-lint та backend-test.

Що відбувається

  1. Docker Buildx — налаштування розширеного білдера з підтримкою кешування
  2. Login до GHCR — автентифікація через GITHUB_TOKEN (автоматичний секрет)
  3. Metadata — генерація тегів: latest + SHA коміту
  4. Build & Push — збірка образу з Dockerfile та публікація

Кешування

Використовується GitHub Actions cache (type=gha) для кешування Docker-шарів між запусками. Це значно прискорює збірку, якщо змінилися тільки верхні шари.

Теги образу

# Приклад тегів:
ghcr.io/username/inventory/backend:latest
ghcr.io/username/inventory/backend:a1b2c3d  # SHA коміту

4.7. docker-frontend Docker

Збирає Docker-образ frontend (multi-stage: Node → Nginx) та публікує в GHCR.

Умова запуску: тільки при push у main, після успішного frontend-build.

Build arguments

Передаються змінні середовища для Vite під час збірки:

АргументЗначенняПризначення
VITE_API_URL /api Базовий URL API
VITE_WS_URL (порожнє) WebSocket URL
VITE_SENTRY_DSN з vars.VITE_SENTRY_DSN Sentry DSN для моніторингу

4.8. deploy Деплой

Автоматичний деплой на продакшн-сервер через SSH після збірки Docker-образів.

Умова запуску: тільки при push у main, після успішного docker-backend та docker-frontend.

Що відбувається на сервері

  1. git pull origin main — оновлення docker-compose.yml та конфігурації
  2. docker compose pull — завантаження нових образів з GHCR
  3. docker compose up -d — перезапуск сервісів
  4. manage.py migrate — виконання міграцій БД
  5. manage.py collectstatic — збір статичних файлів
  6. Health check — перевірка /api/health/
  7. docker image prune — очищення старих образів

Необхідні секрети

СекретОписПриклад
DEPLOY_HOST IP або домен сервера 192.168.1.100
DEPLOY_USER SSH-користувач deploy
DEPLOY_SSH_KEY Приватний SSH-ключ Вміст ~/.ssh/id_ed25519
DEPLOY_PORT SSH-порт (опціонально) 22
DEPLOY_PATH Шлях до проєкту (опціонально) /opt/inventory
Environment: production — deploy використовує GitHub Environment "production". Це дозволяє налаштувати додатковий захист: ручне підтвердження, обмеження по гілках, таймаути. Налаштовується в Settings → Environments.

5. Файли конфігурації

Всі файли, що впливають на роботу CI/CD:

inventory/
  .github/
    workflows/
      ci.yml — головний файл пайплайна
  inventory_project/
    conftest.py — ініціалізація Django для pytest
    pytest.ini — конфігурація pytest
  frontend/
    eslint.config.js — конфігурація ESLint
    tsconfig.json — конфігурація TypeScript
    vite.config.ts — конфігурація Vite
    vitest.config.ts — конфігурація Vitest (якщо окремий)
  setup.cfg — конфігурація flake8, isort, pytest
  requirements.txt — Python залежності

setup.cfg

# Конфігурація pytest
[tool:pytest]
DJANGO_SETTINGS_MODULE = inventory_project.settings
python_files = tests.py test_*.py *_tests.py
python_classes = *Tests Test*
python_functions = test_*

# Конфігурація flake8
[flake8]
max-line-length = 120
exclude = migrations,__pycache__,venv,.venv
extend-ignore = E203

# Конфігурація isort
[isort]
profile = black
skip = migrations
known_django = django
known_drf = rest_framework
sections = FUTURE,STDLIB,THIRDPARTY,DJANGO,DRF,FIRSTPARTY,LOCALFOLDER
line_length = 120

pytest.ini

[pytest]
DJANGO_SETTINGS_MODULE = inventory_project.settings
python_files = tests.py test_*.py *_tests.py
python_classes = *Tests Test*
python_functions = test_*

conftest.py

import os
import django

def pytest_configure():
    os.environ.setdefault("DJANGO_SETTINGS_MODULE", "inventory_project.settings")
    django.setup()

6. Локальний запуск перевірок

Перед push рекомендується запустити ті ж самі перевірки локально, щоб не чекати на CI.

Backend

# Форматування (автовиправлення)
python -m black inventory_project/ --exclude migrations

# Сортування імпортів (автовиправлення)
python -m isort inventory_project/ --skip migrations

# Перевірка стилю (тільки звіт)
python -m flake8 inventory_project/

# Тести з покриттям
cd inventory_project && python -m pytest --cov=inventory -v

Frontend

# Лінтинг (з автовиправленням)
cd frontend && npm run lint -- --fix

# Тести
cd frontend && npx vitest run

# Збірка (перевірка TypeScript + Vite)
cd frontend && npm run build

Все одразу (bash)

#!/bin/bash
# Скрипт для локальної перевірки перед push

echo "=== Backend Lint ==="
python -m black inventory_project/ --exclude migrations
python -m isort inventory_project/ --skip migrations
python -m flake8 inventory_project/ || exit 1

echo "=== Backend Tests ==="
cd inventory_project && python -m pytest -v || exit 1
cd ..

echo "=== Frontend ==="
cd frontend
npm run lint || exit 1
npx vitest run || exit 1
npm run build || exit 1

echo "=== Все перевірки пройшли! ==="

8. Як додати новий Job

Приклад: сповіщення в Slack/Telegram

# Додати в .github/workflows/ci.yml
  notify:
    runs-on: ubuntu-latest
    needs: [deploy]
    if: always()  # навіть якщо деплой впав
    steps:
      - name: Notify Telegram
        uses: appleboy/telegram-action@master
        with:
          to: ${{ secrets.TELEGRAM_CHAT_ID }}
          token: ${{ secrets.TELEGRAM_BOT_TOKEN }}
          message: |
            Deploy ${{ job.status }}: ${{ github.repository }}
            Commit: ${{ github.sha }}
            Author: ${{ github.actor }}

Приклад: перевірка безпеки залежностей

  security-audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - run: pip install pip-audit
      - run: pip-audit -r requirements.txt

9. Секрети та змінні середовища

Поточні змінні (захардкоджені для CI)

Ці значення безпечно використовуються тільки в тестовому оточенні CI:

ЗміннаЗначенняДе використовується
SECRET_KEY test-secret-key-for-ci Django settings
DB_NAME inventory_test PostgreSQL
DB_USER / DB_PASSWORD postgres / postgres PostgreSQL

Секрети для деплою (обов'язкові)

СекретОпис
DEPLOY_HOSTIP-адреса або домен продакшн-сервера
DEPLOY_USERSSH-користувач на сервері
DEPLOY_SSH_KEYПриватний SSH-ключ (вміст файлу, не шлях)
DEPLOY_PORTSSH-порт (за замовчуванням 22)
DEPLOY_PATHШлях до проєкту на сервері (за замовчуванням /opt/inventory)

Як додати секрети

1
Перейдіть в репозиторій на GitHub → Settings → Secrets and variables → Actions
2
Натисніть New repository secret
3
Введіть ім'я (наприклад SSH_PRIVATE_KEY) та значення
4
Використовуйте в workflow: ${{ secrets.SSH_PRIVATE_KEY }}
Ніколи не захардкоджуйте продакшн-секрети (SSH-ключі, API-токени, паролі) безпосередньо у файлі ci.yml. Завжди використовуйте GitHub Secrets.

11. Найкращі практики

Branch Protection Rules

Налаштуйте захист гілки main, щоб заборонити merge без проходження CI:

1
GitHub → Settings → Branches → Add branch protection rule
2
Branch name pattern: main
3
Увімкніть "Require status checks to pass before merging"
4
Додайте CI jobs як обов'язкові: backend-lint, backend-test, frontend-lint, frontend-test, frontend-build

Кешування

Пайплайн вже використовує кешування для прискорення:

  • cache: 'pip' — кешує Python-пакети між запусками
  • cache: 'npm' з cache-dependency-path — кешує node_modules

Рекомендації

  • Завжди запускайте лінтери локально перед push
  • Не ігноруйте помилки CI — виправляйте одразу
  • Пишіть тести для нового функціоналу
  • Тримайте requirements.txt та package.json актуальними
  • Використовуйте develop гілку для розробки, main — для стабільного коду
  • Регулярно оновлюйте версії actions (actions/checkout@v4 тощо)

12. FAQ

Де подивитися результати CI?

GitHub → ваш репозиторій → вкладка Actions. Там видно всі запуски, їх статус та логи кожного кроку.

CI впав, але локально все працює. Чому?

  • Різні версії Python/Node — CI використовує Python 3.12 та Node 20
  • Не закомічені файли — перевірте git status
  • Різні ОС — CI працює на Ubuntu, а ви можете бути на Windows/macOS
  • Кеш — спробуйте очистити кеш в GitHub Actions

Як перезапустити CI без нового коміту?

GitHub → Actions → виберіть workflow run → кнопка "Re-run all jobs" або "Re-run failed jobs".

Як тимчасово вимкнути CI для коміту?

Додайте [skip ci] або [ci skip] в повідомлення коміту:

git commit -m "Оновлення README [skip ci]"

Скільки хвилин GitHub Actions безкоштовно?

Для публічних репозиторіїв — необмежено. Для приватних — 2000 хвилин/місяць на безкоштовному плані.

Як додати покриття коду в PR?
