1. Схема деплою

Проєкт підтримує два способи деплою:

Ручний деплой
Запуск .\deploy.ps1 на сервері.
Збирає образи локально, виконує міграції, перезапускає сервіси.
Автоматичний деплой
Push у main → GitHub Actions → CI перевірки → збірка Docker-образів → SSH на сервер → оновлення.

Мережева конфігурація

СервісПорт на серверіДоступ
Додаток (Nginx)8081Зовнішній (з інтернету)
PostgreSQL5433Тільки локально
Redis6380Тільки локально
OpenSSH22Зовнішній (для CD)
Доступ з інтернету — щоб додаток був доступний ззовні, потрібно:
  • Відкрити порт 8081 у брандмауері Windows
  • Налаштувати проброс портів (port forwarding) на роутері: зовнішній порт → IP сервера:8081
  • Або використати статичний IP / DDNS-сервіс, якщо IP динамічний

3. Налаштування OpenSSH Server

OpenSSH потрібен для автоматичного деплою через GitHub Actions. Якщо ви плануєте використовувати тільки ручний деплой — цей крок можна пропустити.

Крок 1: Встановлення OpenSSH Server

PowerShell (Адміністратор)
# Перевірити чи встановлений
Get-WindowsCapability -Online | Where-Object Name -like 'OpenSSH.Server*'

# Встановити якщо не встановлений
Add-WindowsCapability -Online -Name OpenSSH.Server~~~~0.0.1.0

Крок 2: Запуск та автостарт сервісу

PowerShell (Адміністратор)
# Запустити сервіс
Start-Service sshd

# Автозапуск при старті Windows
Set-Service -Name sshd -StartupType Automatic

# Перевірити статус
Get-Service sshd

# Очікуваний результат:
# Status   Name    DisplayName
# ------   ----    -----------
# Running  sshd    OpenSSH SSH Server

Крок 3: Зміна стандартної оболонки на Git Bash

GitHub Actions SSH-action відправляє bash-команди. Потрібно зробити Git Bash оболонкою за замовчуванням для SSH:

PowerShell (Адміністратор)
# Встановити Git Bash як оболонку за замовчуванням для SSH
New-ItemProperty -Path "HKLM:\SOFTWARE\OpenSSH" `
    -Name DefaultShell `
    -Value "C:\Program Files\Git\bin\bash.exe" `
    -PropertyType String -Force

# Перезапустити SSH-сервіс
Restart-Service sshd
Важливо! Якщо Git встановлений в іншій директорії, замініть шлях. Перевірте: where.exe bash

Крок 4: Генерація SSH-ключа для GitHub Actions

PowerShell
# Спочатку створіть папку .ssh (якщо її ще немає)
mkdir $env:USERPROFILE\.ssh

# Генеруємо нову пару ключів для деплою
ssh-keygen -t ed25519 -C "github-actions-deploy" -f $env:USERPROFILE\.ssh\deploy_key

# Коли з'явиться запит "Enter passphrase" — натисніть Enter двічі (порожній пароль).
# УВАГА: Параметр -N "" НЕ працює на Windows OpenSSH!

# Результат — два файли:
#   ~/.ssh/deploy_key       — ПРИВАТНИЙ ключ (додати в GitHub Secrets)
#   ~/.ssh/deploy_key.pub   — ПУБЛІЧНИЙ ключ (додати в authorized_keys)

Крок 5: Додати публічний ключ до authorized_keys

PowerShell
# Для звичайного користувача:
Get-Content $env:USERPROFILE\.ssh\deploy_key.pub | Add-Content $env:USERPROFILE\.ssh\authorized_keys

# Для адміністратора (Windows використовує інший файл!):
Get-Content $env:USERPROFILE\.ssh\deploy_key.pub | Add-Content $env:ProgramData\ssh\administrators_authorized_keys

# Встановити правильні права (для адміністратора):
# Використовуємо SID замість імен груп (працює на будь-якій мові Windows):
icacls $env:ProgramData\ssh\administrators_authorized_keys /inheritance:r /grant "*S-1-5-18:(F)" /grant "*S-1-5-32-544:(F)"

# S-1-5-18     = SYSTEM
# S-1-5-32-544 = Administrators (BUILTIN\Administrators)
Який файл використовувати?
  • Якщо SSH-користувач — адміністратор → C:\ProgramData\ssh\administrators_authorized_keys
  • Якщо SSH-користувач — звичайний → C:\Users\USERNAME\.ssh\authorized_keys

Крок 6: Перевірка SSH-з'єднання

PowerShell
# Перевірка з'єднання з локальної машини
ssh -i $env:USERPROFILE\.ssh\deploy_key localhost "echo SSH працює && docker --version"

# Очікуваний результат:
# SSH працює
# Docker version 27.x.x, build xxxxxxx

Крок 7: Скопіювати приватний ключ

Скопіюйте вміст приватного ключа — він знадобиться для GitHub Secrets:

PowerShell
# Вивести вміст приватного ключа
Get-Content $env:USERPROFILE\.ssh\deploy_key

# Скопіюйте ВСЕ, включаючи рядки:
# -----BEGIN OPENSSH PRIVATE KEY-----
# ...
# -----END OPENSSH PRIVATE KEY-----

4. Налаштування брандмауера

Відкрити порти

PowerShell (Адміністратор)
# Порт 8081 — веб-додаток (доступ з інтернету)
New-NetFirewallRule -DisplayName "IT Inventory Web" `
    -Direction Inbound -Protocol TCP -LocalPort 8081 `
    -Action Allow -Profile Any

# Порт 22 — SSH (для автодеплою з GitHub Actions)
New-NetFirewallRule -DisplayName "OpenSSH Server" `
    -Direction Inbound -Protocol TCP -LocalPort 22 `
    -Action Allow -Profile Any
Порт 22 (SSH) — якщо не плануєте автодеплой через GitHub Actions, цей порт відкривати не потрібно.

Port Forwarding на роутері

Щоб додаток був доступний з інтернету, налаштуйте проброс портів на роутері:

Зовнішній портВнутрішній IPВнутрішній портПротокол
8081IP вашого сервера (наприклад 192.168.1.100)8081TCP
22IP вашого сервера22TCP

Дізнатись свій зовнішній IP

PowerShell
# Зовнішній IP (для GitHub Secrets)
Invoke-RestMethod http://ifconfig.me/ip

# Локальний IP (для port forwarding)
Get-NetIPAddress -AddressFamily IPv4 | Where-Object { $_.InterfaceAlias -notlike "*Loopback*" }
Динамічний IP? Якщо ваш провайдер змінює IP, використайте безкоштовний DDNS-сервіс (No-IP, DuckDNS тощо). Тоді в GitHub Secrets замість IP вкажете доменне ім'я.

7. Ручний деплой (deploy.ps1)

Скрипт deploy.ps1 автоматизує весь процес оновлення:

Базовий запуск

PowerShell
# Стандартний деплой: git pull + build + up + migrate
.\deploy.ps1

Параметри

ПараметрОписПриклад
-SkipPull Не виконувати git pull .\deploy.ps1 -SkipPull
-SkipMigrate Не виконувати міграції БД .\deploy.ps1 -SkipMigrate
-SkipBuild Не збирати образи (docker pull замість build) .\deploy.ps1 -SkipBuild
-Force Повністю зупинити і перезапустити всі контейнери .\deploy.ps1 -Force

Комбінації параметрів

PowerShell
# Тільки перезапуск без оновлення коду
.\deploy.ps1 -SkipPull -SkipBuild

# Повне перерозгортання з нуля
.\deploy.ps1 -Force

# Швидке оновлення (тільки код, без міграцій)
.\deploy.ps1 -SkipMigrate

8. Автоматичний деплой (GitHub Actions)

При кожному push у main GitHub Actions автоматично:

  1. Запускає CI-перевірки (лінтинг, тести, збірка)
  2. Збирає Docker-образи та публікує в GHCR
  3. Підключається до вашого сервера через SSH
  4. Виконує git pull + docker compose up -d + міграції

Що відбувається на сервері (SSH-скрипт)

# GitHub Actions виконує ці команди через SSH:
cd /d/inventory          # Git Bash шлях до D:\inventory
git pull origin main     # Оновлення коду
docker compose pull      # Завантаження нових образів
docker compose up -d     # Перезапуск сервісів
# ... міграції, collectstatic, health check

Потік автодеплою

git push origin main
      │
      ▼
┌─────────────────────┐
│  GitHub Actions CI   │  Лінтинг, тести, збірка
│  (~3-5 хвилин)       │
└──────────┬──────────┘
           │ Все пройшло ✓
           ▼
┌─────────────────────┐
│  Docker Build & Push │  Збірка образів → GHCR
│  (~3-5 хвилин)       │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│  SSH → Ваш сервер    │  pull + up + migrate
│  (~1 хвилина)        │
└──────────┬──────────┘
           │
           ▼
     Додаток оновлено!

9. Налаштування GitHub Secrets

Покрокова інструкція

1
Перейдіть у ваш репозиторій на GitHub (наприклад, https://github.com/YOUR_USERNAME/inventory)
2
Натисніть вкладку Settings (верхнє меню репозиторію, праворуч)
3
У лівій бічній панелі знайдіть розділ "Security" і натисніть Secrets and variables → Actions
4
Натисніть зелену кнопку "New repository secret"
5
Заповніть поля:
• Name — ім'я секрету (наприклад, DEPLOY_HOST)
• Secret — значення секрету (наприклад, 91.222.154.123)
Натисніть "Add secret"
6
Повторіть кроки 4-5 для кожного секрету з таблиці нижче

Список секретів (додайте всі 4)

Ім'я секрету (Name)Що вписати (Secret)Як отримати значення
DEPLOY_HOST Ваш зовнішній IP-адреса
Приклад: 91.222.154.123		 Invoke-RestMethod http://ifconfig.me/ip
DEPLOY_USER Ім'я Windows-користувача
Приклад: Администратор		 $env:USERNAME
DEPLOY_SSH_KEY Весь вміст приватного ключа
(багаторядковий текст)		 Див. інструкцію нижче
DEPLOY_PORT 22 Стандартний SSH-порт

Як скопіювати SSH-ключ для DEPLOY_SSH_KEY

На сервері виконайте команду:

PowerShell (на сервері)
# Вивести вміст приватного ключа
Get-Content $env:USERPROFILE\.ssh\deploy_key

У терміналі з'явиться текст приблизно такого вигляду:

-----BEGIN OPENSSH PRIVATE KEY-----
b3BlbnNzaC1rZXktdjEAAAAABG5vbmUAAAAEbm9uZQAAAAAAAAABAAAAMwAAAAtzc2gtZW
QyNTUxOQAAACBxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx...
...кілька рядків...
xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx==
-----END OPENSSH PRIVATE KEY-----
ВАЖЛИВО! Скопіюйте ВЕСЬ текст — від -----BEGIN OPENSSH PRIVATE KEY----- до -----END OPENSSH PRIVATE KEY----- включно. Вставте його в поле Secret при створенні секрету DEPLOY_SSH_KEY.
Як правильно скопіювати:
  1. Виділіть весь текст ключа в терміналі (від -----BEGIN до -----END...-----)
  2. Натисніть Ctrl+C для копіювання
  3. Перейдіть у GitHub → New repository secret
  4. В полі Name введіть: DEPLOY_SSH_KEY
  5. В полі Secret натисніть Ctrl+V для вставки
  6. Натисніть "Add secret"

Після додавання всіх 4 секретів, сторінка Settings → Secrets → Actions повинна виглядати так:

12. Проблеми та рішення

Контейнери не запускаються

PowerShell
# Подивитись логи
docker compose logs --tail 100

# Перевірити чи Docker запущений
docker info

# Перезапустити Docker Desktop
# Правою кнопкою на значок Docker в треї → Restart

Порт 8081 зайнятий

PowerShell
# Хто використовує порт 8081
netstat -ano | findstr :8081

# Змінити порт в .env:
# APP_PORT=8082
# Потім:
docker compose down && docker compose up -d

SSH не працює (GitHub Actions)

  • Перевірте що OpenSSH Server запущений: Get-Service sshd
  • Перевірте що порт 22 відкритий у брандмауері
  • Перевірте що Git Bash встановлений як default shell
  • Перевірте що публічний ключ додано в правильний authorized_keys файл
  • Перевірте що port forwarding налаштований на роутері

GitHub Actions deploy job скіпається

Deploy запускається тільки при push у main. Перевірте:

  • Push зроблений саме в main, не в іншу гілку
  • Це push, а не pull_request
  • Всі CI-перевірки пройшли успішно

Міграції не працюють

PowerShell
# Перевірити з'єднання з БД
docker compose exec web python inventory_project/manage.py dbshell

# Статус міграцій
docker compose exec web python inventory_project/manage.py showmigrations

Не вистачає місця на диску

PowerShell
# Очистити невикористані Docker-ресурси
docker system prune -a --volumes

# Тільки образи

Динамічний IP змінився

Якщо ваш зовнішній IP змінився, оновіть:

  1. DEPLOY_HOST в GitHub Secrets
  2. ALLOWED_HOSTS в .env на сервері
  3. Перезапустіть: docker compose restart web
Порада: Використайте DDNS-сервіс (наприклад, DuckDNS — безкоштовний), щоб не оновлювати IP вручну.
